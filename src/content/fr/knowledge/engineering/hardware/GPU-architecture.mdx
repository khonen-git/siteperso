## Compute Core

Le **cœur de calcul** regroupe les unités qui exécutent les kernels CUDA. Sur NVIDIA, la hiérarchie descend du chip entier vers les unités élémentaires de traitement.

### GPC et TPC

Un **GPC** (*Graphics Processing Cluster*) est un ensemble de ressources de calcul partiellement indépendantes. Chaque GPC contient plusieurs **TPC** (*Texture Processing Clusters*), qui regroupent à leur tour des **SM**.

Cette organisation permet de paralléliser massivement le travail : un kernel lancé sur le GPU est réparti sur l'ensemble des GPC disponibles.

### SM — Streaming Multiprocessor

### Cache L2

Le **cache L2** est partagé entre tous les SM du GPU. Il sert de tampon entre la VRAM et les caches L1, et joue un rôle clé dans les accès mémoire répétitifs (convolutions, réductions). Sa taille varie selon le modèle (de quelques Mo à plus de 90 Mo sur les GPU datacenter récents).

**Hiérarchie mémoire CUDA** (du plus rapide au plus lent) : registres → shared memory → L1 → L2 → VRAM → mémoire hôte (RAM CPU via PCIe).

## Data Transfer

Les transferts entre CPU et GPU passent par le bus **PCIe** (généralement PCIe 4.0 x16 ou PCIe 5.0 sur les plateformes récentes). Le débit typique est de l'ordre de 25–60 Go/s, bien inférieur à la bande passante interne du GPU.

- **Resizable BAR** (*Base Address Register*) : permet au CPU d'accéder à l'intégralité de la VRAM sans fenêtre mémoire limitée. Utile pour certains workloads, mais l'impact en deep learning est modéré.
- **Copy Engines** (*DMA*) : moteurs de copie asynchrones dédiés qui transfèrent des données en parallèle du calcul, sans bloquer les SM.

En pratique, on minimise les transferts CPU↔GPU en gardant les tenseurs sur le device (`tensor.cuda()` ou `.to("cuda")`).

### Occupancy

L'**occupancy** mesure le ratio de warps actifs par rapport au maximum possible sur un SM. Elle dépend du nombre de registres par thread, de la shared memory utilisée et de la taille des blocks. Une occupancy élevée masque la latence mémoire, mais n'est pas toujours synonyme de performance maximale.

### Exemple minimal CUDA

<CodeBlock language="cuda">
{`__global__ void saxpy(float *x, float *y, float a, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n)
        y[i] = a * x[i] + y[i];
}

// Lancement : grid de blocks, chaque block de 256 threads
saxpy<<<(n + 255) / 256, 256>>>(d_x, d_y, 2.0f, n);`}
</CodeBlock>

En PyTorch, ce niveau bas n'est en général pas nécessaire : les opérations tensorielles appellent automatiquement cuBLAS/cuDNN.

## Multi-GPU

Pour scaler au-delà d'un seul GPU :

- **NVLink** : interconnexion haute bande passante entre GPU NVIDIA (jusqu'à plusieurs centaines de Go/s). Disponible sur les GPU datacenter (A100, H100), absent sur la plupart des GPU grand public.
- **NCCL** (*NVIDIA Collective Communications Library*) : bibliothèque de communication collective (all-reduce, broadcast) optimisée pour le multi-GPU. Utilisée en interne par PyTorch Distributed et Horovod.

Stratégies courantes en deep learning :
- **Data Parallelism** (DDP) : chaque GPU traite un sous-batch, les gradients sont synchronisés via all-reduce.
- **Model Parallelism** : différentes couches ou shards du modèle sur différents GPU (nécessaire pour les très grands modèles).

## Points clés pour la finance quantitative

<MdxCard>
- **Entraînement de modèles** (LSTM, Transformers, XGBoost GPU via cuML) : les Tensor Cores et cuDNN sont déterminants.
- **Inférence temps réel** (scoring, signaux) : la latence PCIe et la taille du batch comptent autant que le débit brut.
- **Backtesting vectorisé** : CuPy ou RAPIDS peuvent accélérer le traitement de grandes séries temporelles.
- **Profiling** : utiliser `torch.profiler` ou NVIDIA Nsight Systems pour identifier les goulots (compute-bound vs memory-bound).
- **VRAM** : surveiller `torch.cuda.max_memory_allocated()` — un OOM est souvent le premier signal qu'il faut réduire le batch ou activer le gradient checkpointing.
</MdxCard>
