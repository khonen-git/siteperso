## Compute Core

Le **cœur de calcul** exécute les instructions du programme. Contrairement au GPU (SIMT, milliers de threads légers), le CPU est optimisé pour la **latence** et les branches conditionnelles complexes — typiques du code Python métier en finance.

### Cœurs physiques et threads logiques

Un **cœur physique** est une unité d'exécution indépendante. Le **SMT** (*Simultaneous Multi-Threading*, Hyper-Threading chez Intel) expose deux **threads logiques** par cœur, partageant les unités d'exécution. En quant, cela aide surtout les workloads mixtes (I/O + calcul), mais pour du calcul pur (NumPy, BLAS), compter les **cœurs physiques** est souvent plus fiable que le nombre de threads logiques.

### SIMD et vectorisation

Les instructions **SSE**, **AVX** et **AVX-512** permettent d'exécuter des opérations sur des registres larges (128, 256 ou 512 bits). En pratique, la plupart des bibliothèques scientifiques Python (NumPy, SciPy, scikit-learn via BLAS) exploitent déjà ces instructions sous le capot — d'où l'importance d'installer une build **MKL** ou **OpenBLAS** optimisée.

## Cache Hierarchy

## I/O and Interconnect

Le CPU orchestre les échanges avec le disque, le réseau et le **GPU** via **PCIe**. En pipeline quant typique :

- **Lecture disque** → CPU (pandas/Polars) → feature engineering → entraînement GPU ou CPU
- Le CPU reste le goulot pour l'ingestion de ticks, le parsing CSV/Parquet et la logique métier Python

Les liaisons **QPI/UPI** (Intel) ou **Infinity Fabric** (AMD) connectent les cœurs entre eux et vers le contrôleur mémoire.

### Le GIL et ses implications

Le **GIL** (*Global Interpreter Lock*) de CPython empêche l'exécution parallèle de bytecode Python sur plusieurs cœurs. En pratique :

- **Calcul NumPy/BLAS** : le GIL est libéré → parallélisme multi-cœur effectif
- **Boucles Python pures** : un seul cœur → utiliser `multiprocessing` ou réécrire en NumPy/Cython/Numba
- **I/O** (lecture fichiers, API) : `threading` suffit car le GIL est libéré pendant l'attente

### Exemple : paralléliser un backtest

<CodeBlock language="python">
{`from joblib import Parallel, delayed
import numpy as np

def backtest_strategy(returns: np.ndarray, threshold: float) -> float:
  signals = returns > threshold
  return (signals * returns).sum()

returns_matrix = np.random.randn(100, 252)  # 100 stratégies, 252 jours

results = Parallel(n_jobs=-1)(
    delayed(backtest_strategy)(returns_matrix[i], 0.01)
    for i in range(100)
)`}
</CodeBlock>

`n_jobs=-1` utilise tous les cœurs disponibles via des processus séparés.

## Multi-Socket et NUMA

Sur les serveurs de production (calcul de risque, Monte Carlo massif), on trouve souvent **2–4 sockets** CPU. Chaque socket a sa mémoire locale :

- Placer les workers sur le même socket que leurs données (`numactl --cpunodebind`)
- Éviter le **false sharing** : deux threads modifiant des variables adjacentes dans le cache
- Pour Polars/NumPy multi-threadés, limiter `OMP_NUM_THREADS` au nombre de cœurs d'un socket si NUMA

## Points clés pour la finance quantitative

<MdxCard>
- **Choisir le bon parallélisme** : `threading` pour l'I/O, `multiprocessing`/`joblib` pour le Python pur, BLAS multi-thread pour l'algèbre linéaire.
- **Limiter OMP_NUM_THREADS** : si vous lancez 8 workers `joblib` et que chaque worker lance 8 threads BLAS, vous saturez le CPU (oversubscription).
- **Préférer Polars** pour les gros DataFrames et les agrégations complexes.
- **Profiler avant d'optimiser** : `cProfile`, `py-spy` ou `scalene` pour distinguer temps Python vs temps C/BLAS.
- **RAM** : un historique tick-by-tick sur 5000 actions peut dépasser 64 Go — planifier le partitionnement (Parquet par date/symbole) dès le départ.
- **Le CPU reste central** : même dans un pipeline GPU, l'ingestion, le preprocessing et l'orchestration tournent sur CPU.
</MdxCard>
