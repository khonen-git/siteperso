## Compute Core

The **compute core** groups the units that execute CUDA kernels. On NVIDIA GPUs, the hierarchy runs from the full chip down to the elementary processing units.

### GPC and TPC

A **GPC** (*Graphics Processing Cluster*) is a set of partially independent compute resources. Each GPC contains several **TPCs** (*Texture Processing Clusters*), which in turn group **SMs**.

This organization enables massive parallelism: a kernel launched on the GPU is distributed across all available GPCs.

### L2 Cache

The **L2 cache** is shared across all SMs on the GPU. It acts as a buffer between VRAM and L1 caches, and plays a key role in repetitive memory accesses (convolutions, reductions). Its size varies by model (from a few MB to over 90 MB on recent datacenter GPUs).

**CUDA memory hierarchy** (fastest to slowest): registers → shared memory → L1 → L2 → VRAM → host memory (CPU RAM via PCIe).

## Data Transfer

Transfers between CPU and GPU go through the **PCIe** bus (typically PCIe 4.0 x16 or PCIe 5.0 on recent platforms). Typical throughput is on the order of 25–60 GB/s, well below the GPU's internal bandwidth.

- **Resizable BAR** (*Base Address Register*): allows the CPU to access the full VRAM without a limited memory window. Useful for some workloads, but impact in deep learning is moderate.
- **Copy Engines** (*DMA*): dedicated asynchronous copy engines that transfer data in parallel with computation, without blocking SMs.

In practice, CPU↔GPU transfers are minimized by keeping tensors on the device (`tensor.cuda()` or `.to("cuda")`).

### Occupancy

**Occupancy** measures the ratio of active warps to the maximum possible on an SM. It depends on registers per thread, shared memory usage, and block size. High occupancy masks memory latency, but is not always synonymous with maximum performance.

## Multi-GPU

To scale beyond a single GPU:

- **NVLink**: high-bandwidth interconnect between NVIDIA GPUs (up to several hundred GB/s). Available on datacenter GPUs (A100, H100), absent on most consumer GPUs.
- **NCCL** (*NVIDIA Collective Communications Library*): collective communication library (all-reduce, broadcast) optimized for multi-GPU. Used internally by PyTorch Distributed and Horovod.

Common deep learning strategies:
- **Data Parallelism** (DDP): each GPU processes a sub-batch, gradients are synchronized via all-reduce.
- **Model Parallelism**: different layers or model shards on different GPUs (required for very large models).

## Key points for quantitative finance

<MdxCard>
- **Model training** (LSTM, Transformers, XGBoost GPU via cuML): Tensor Cores and cuDNN are decisive.
- **Real-time inference** (scoring, signals): PCIe latency and batch size matter as much as raw throughput.
- **Vectorized backtesting**: CuPy or RAPIDS can accelerate processing of large time series.
- **Profiling**: use `torch.profiler` or NVIDIA Nsight Systems to identify bottlenecks (compute-bound vs memory-bound).
- **VRAM**: monitor `torch.cuda.max_memory_allocated()` — an OOM is often the first signal to reduce batch size or enable gradient checkpointing.
