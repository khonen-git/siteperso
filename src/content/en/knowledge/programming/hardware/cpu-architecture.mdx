## Compute Core

The **compute core** executes program instructions. Unlike the GPU (SIMT, thousands of lightweight threads), the CPU is optimized for **latency** and complex conditional branches — typical of business logic Python code in finance.

### Physical cores and logical threads

A **physical core** is an independent execution unit. **SMT** (*Simultaneous Multi-Threading*, Hyper-Threading on Intel) exposes two **logical threads** per core, sharing execution units. In quant workloads, this mainly helps mixed I/O + compute tasks; for pure compute (NumPy, BLAS), counting **physical cores** is often more reliable than logical thread count.

### SIMD and vectorization

**SSE**, **AVX**, and **AVX-512** instructions operate on wide registers (128, 256, or 512 bits). In practice, most scientific Python libraries (NumPy, SciPy, scikit-learn via BLAS) already exploit these instructions under the hood — which is why installing an optimized **MKL** or **OpenBLAS** build matters.

## I/O and Interconnect

The CPU orchestrates exchanges with disk, network, and **GPU** via **PCIe**. In a typical quant pipeline:

- **Disk read** → CPU (pandas/Polars) → feature engineering → GPU or CPU training
- The CPU remains the bottleneck for tick ingestion, CSV/Parquet parsing, and Python business logic

**QPI/UPI** (Intel) or **Infinity Fabric** (AMD) links connect cores to each other and to the memory controller.

### The GIL and its implications

CPython's **GIL** (*Global Interpreter Lock*) prevents parallel execution of Python bytecode across multiple cores. In practice:

- **NumPy/BLAS compute**: GIL is released → effective multi-core parallelism
- **Pure Python loops**: single core → use `multiprocessing` or rewrite in NumPy/Cython/Numba
- **I/O** (file reads, APIs): `threading` is sufficient since the GIL is released during waits

### Example: parallelizing a backtest

<CodeBlock language="python">
{`from joblib import Parallel, delayed
import numpy as np

def backtest_strategy(returns: np.ndarray, threshold: float) -> float:
  signals = returns > threshold
  return (signals * returns).sum()

returns_matrix = np.random.randn(100, 252)  # 100 strategies, 252 days

results = Parallel(n_jobs=-1)(
    delayed(backtest_strategy)(returns_matrix[i], 0.01)
    for i in range(100)
)`}
</CodeBlock>

`n_jobs=-1` uses all available cores via separate processes.

## Multi-Socket and NUMA

Production servers (risk computation, massive Monte Carlo) often have **2–4 CPU sockets**. Each socket has local memory:

- Place workers on the same socket as their data (`numactl --cpunodebind`)
- Avoid **false sharing**: two threads modifying adjacent variables in cache
- For multi-threaded Polars/NumPy, limit `OMP_NUM_THREADS` to one socket's core count on NUMA systems

## Key points for quantitative finance

<MdxCard>
- **Choose the right parallelism**: `threading` for I/O, `multiprocessing`/`joblib` for pure Python, multi-thread BLAS for linear algebra.
- **Limit OMP_NUM_THREADS**: if you launch 8 `joblib` workers and each spawns 8 BLAS threads, you oversubscribe the CPU.
- **Prefer Polars** for large DataFrames and complex aggregations.
- **Profile before optimizing**: `cProfile`, `py-spy`, or `scalene` to distinguish Python time vs C/BLAS time.
- **RAM**: tick-by-tick history on 5000 stocks can exceed 64 GB — plan partitioning (Parquet by date/symbol) from the start.
- **CPU remains central**: even in a GPU pipeline, ingestion, preprocessing, and orchestration run on CPU.
</MdxCard>
